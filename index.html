MODULE 01 / 06

How Infrastructure Actually Gets Deployed Today

The mental model every technical prospect carries. Learn to map what they say to a diagnostic framework — and use it on every call.

⏱ ~8 min9 sectionsKnowledge check
01The Simple Mental Model
▼

Every infrastructure change follows the same pattern. Once you know this flow, you can map anything a prospect says back to it.

Code
→
Git
→
Pipeline
→
Cloud
  • Code — defines what should exist (Terraform / OpenTofu)
  • Git — where changes are proposed (pull requests / commits)
  • Pipeline — where changes are executed (CI/CD tools)
  • Cloud — where infrastructure actually runs (AWS / Azure / GCP)
🔎 What Prospects Say
"We run Terraform through GitHub Actions"
"Changes go through pull requests"
"We deploy through Jenkins"

These are all fragments of the same system. Your job is to translate them into the full flow.

02Translating What Prospects Say
▼

Prospects describe tools, not workflows. Translate automatically.

Tool → Flow Stage

  • "GitHub Actions" / "Jenkins" → Pipeline
  • "Pull requests" / "PRs" → Git
  • "Terraform" → Code
  • "AWS / Azure / GCP" → Cloud

Why It Matters

Strong reps don't get stuck on tools. They map to the flow — and that's where real problems live.

03Why This Model Matters on Calls
▼

Every problem you'll hear maps somewhere in this chain. When a prospect describes pain, ask: "Where in the flow is this breaking?"

  • Visibility issues → Pipeline / Cloud
  • Approval issues → Pipeline
  • Infrastructure drift → Code vs. Cloud mismatch
  • Coordination issues → Across multiple stages
04Early Stage vs. At Scale
▼

Early Stage

  • Simple workflows, few engineers
  • Minimal coordination needed
  • Pipeline works fine

At Scale

  • Multiple teams, more environments
  • More coordination required
  • Higher risk of mistakes
Key Insight

The flow stays the same. What changes is complexity, number of people involved, and risk of things going wrong.

05Where Things Start to Break
▼

At scale, problems appear in predictable ways — all breakdowns in the flow:

  • Pipelines become harder to manage
  • Approvals become manual and inconsistent
  • Visibility into changes disappears
  • Teams start working differently from each other
  • Infrastructure drifts from what's in code
06Pattern Recognition Framework
▼
  • Hear the signal — "We run Terraform through GitHub Actions"
  • Map to the flow — Terraform = Code, GitHub Actions = Pipeline
  • Anticipate risk areas — Is this standardized? How are approvals handled?
  • Ask the follow-up — "Is that workflow consistent across teams?"
07What Good Discovery Sounds Like
▼

✗ Weak

"What tools are you using?"

✓ Strong

"Can you walk me through how infrastructure changes actually get deployed today?"

Then map their answer to the flow. The strong approach gets you a system description — the weak one gets you a tool list.

08Rep Listening Guide
▼

Best Follow-Ups

  • How does a change move from code to production?
  • Is that consistent across teams?
  • Where does it tend to break down?

What NOT To Do

  • Get stuck in tool details
  • Assume the workflow is clean
  • Skip understanding the full flow

MODULE 02 / 06

Terraform in the Real World — and Where It Breaks

A pattern recognition module for real-world infrastructure pain. Not a Terraform tutorial — a field guide for what breaks and why.

⏱ ~8 min6 sectionsKnowledge check
01What Terraform Does — and Doesn't Do
▼

Terraform defines infrastructure as code and applies changes to cloud environments. It answers "what should exist" and "what should change."

✓ Terraform handles

  • Defining desired state
  • Planning and applying changes
  • Tracking what exists today

✗ Terraform does NOT handle

  • Who can make changes
  • Cross-team coordination
  • Approval workflows
  • Policy enforcement at scale
  • Central audit and tracking
Key Takeaway

"We use Terraform" tells you they've standardized on IaC. It tells you nothing about whether their workflows scale, governance exists, or teams are coordinated.

🔎 Early Warning Signals
"It works, but it's messy"
"Every team does it a little differently"
02How Terraform Is Actually Run
▼

Most teams run Terraform through general-purpose CI/CD pipelines. Works early. Breaks at scale.

Code
→
Git
→
GitHub Actions / Jenkins
→
Cloud
The Core Mismatch

CI/CD was built for application delivery. Infrastructure has different requirements: long-lived state, cross-system dependencies, destructive change risk, multi-team coordination. Using CI/CD for both creates technical debt that grows with scale.

03The Four Breaking Points
▼

Terraform setups break in four predictable ways as teams scale. Learn all four.

1. Visibility Breaks

🔎 Signals
"We don't always know what changed"
"Hard to trace changes across environments"

Impact: Slower debugging, higher incident risk. Ask: "If something breaks in production, how do you trace what changed?"

2. Approvals & Coordination Break

🔎 Signals
"Approvals are handled in Slack"
"Prod changes require manual coordination"

Impact: Risk of bad changes reaching production, slower deployments. Ask: "How are production changes approved today? Is that consistent across teams?"

3. Ownership Breaks

🔎 Signals
"Every team has its own process"
"It's unclear who owns what"

Impact: Confusion during incidents, slower onboarding. Ask: "Is your Terraform workflow standardized across teams?"

4. Secrets & Access Break

🔎 Signals
"Security is concerned about credentials"
"Access is inconsistent across teams"

Impact: Increased risk exposure, audit challenges. Ask: "How are credentials handled during Terraform runs?"

04Pattern Recognition Framework
▼
  • Hear the signal — "We run Terraform through GitHub Actions and it mostly works"
  • Interpret — CI/CD used as orchestration; standardization may be weak
  • Expand — "Is this consistent across teams? Where does it break down?"
  • Identify type — Coordination? Governance? Security?
  • Light positioning (only after confirming pain) — "Most teams at that stage run into coordination challenges since CI/CD wasn't built for infrastructure workflows."
05What Good Discovery Sounds Like
▼

✗ Weak

"We help manage Terraform workflows."

✓ Strong

"How is Terraform actually being run across teams today? Where does the process start to break down?"

The Reframe

"It sounds like the issue isn't Terraform itself — it's how the workflow is coordinated across teams."

MODULE 03 / 06

Why Cloud Complexity Becomes a Problem

Nothing is broken yet — but everything is getting harder. Learn to recognize the signals that systems are outgrowing their original design.

⏱ ~8 min8 sectionsKnowledge check
01The Core Insight — Cloud Starts Simple
▼

Early-stage infrastructure is manageable by design. One team, a few services, one or two environments, a simple Terraform workflow. Everything fits in your head.

Early Stage

  • One team, few services
  • One or two environments
  • Simple Terraform workflow
  • Everything feels manageable

What Changes Over Time

  • More teams added
  • More services created
  • More environments appear
  • More dependencies form
Key Takeaway

The technology doesn't fundamentally change. The number of interactions between systems and teams increases. That's what creates complexity.

🔎 Early Scale Signals
"Things are getting harder to manage"
"We have more teams now"
"It worked fine before, but now it's getting messy"
02Where Complexity Comes From
▼

Complexity grows in four predictable ways. Learn to recognize all of them.

1. Environments Multiply

🔎 Signals
"We have dev, staging, prod… and more"
"Each team has their own environments"

What it means: More places to manage infrastructure, increased chance of inconsistency between environments.

2. Access and Permissions Expand

🔎 Signals
"Access is different across teams"
"Permissions are getting harder to manage"

What it means: More people interacting with infrastructure, more risk of misconfiguration.

3. Shared Resources Increase

🔎 Signals
"Multiple teams depend on the same systems"
"Changes in one place affect others"

What it means: Hidden dependencies between systems. One team's change can break another team's environment.

4. Dependencies Become Critical

🔎 Signals
"Changes have to happen in a certain order"
"We've had issues with things breaking across services"

What it means: Sequencing matters and nothing is managing it cleanly. This is a coordination problem, not just a tooling issue.

03What Complexity Feels Like
▼

By the time a prospect describes these signals, the system has already outgrown its original design. They may not have diagnosed the cause — but the symptoms are clear.

  • "Things are getting harder to manage"
  • "We have to coordinate across teams more and more"
  • "Deployments are more fragile than they used to be"
  • "We're seeing more issues as we scale"
Business Impact

Slower delivery, increased risk of failure, more operational overhead. These aren't random — they're the direct consequence of coordination debt accumulating.

04Why CI/CD + Terraform Breaks at Scale
▼

Early Stage

  • Pipeline runs Terraform cleanly
  • Changes are simple and isolated
  • Few dependencies to manage

At Scale — What Breaks

  • Pipelines don't understand dependencies
  • No coordination across teams
  • Workflows become fragmented
  • Each team builds its own solution
🔎 Fragmentation Signals
"Pipelines are getting hard to manage"
"Each team built their own workflow"
"Things aren't consistent anymore"
Key Takeaway

CI/CD tools are not designed to manage cross-team workflows, infrastructure dependencies, or complex coordination. When a prospect describes fragmentation, orchestration is what's missing.

05Pattern Recognition Framework
▼
  • Hear the signal — "We have multiple teams and environments, and things are getting harder to manage"
  • Interpret — Complexity has increased; coordination is now a problem
  • Expand — "Where do things break down most often? Which teams need to coordinate changes?"
  • Identify type — Coordination problem? Dependency problem? Workflow fragmentation?
  • Light positioning (only after confirming pain) — "Most teams at that stage run into coordination issues as complexity grows, especially when multiple teams and dependencies are involved."
06What Good Discovery Sounds Like
▼

✗ Weak

"Are you using multiple clouds?"

✓ Strong

"How many teams are working on infrastructure today? How do changes get coordinated across systems? Where does the process start to break down?"

The Reframe

"It sounds like the challenge isn't a specific tool — it's the growing complexity and coordination across teams."

▼

MODULE 04 / 06

Governance, Control, and Risk

This is no longer just an engineering problem — it's a business risk problem. Learn to recognize governance signals and find the budget behind them.

⏱ ~8 min8 sectionsKnowledge check
01What Governance Actually Means
▼

When a prospect says "we need better governance," they're describing one of four specific problems. Your job is to find out which one.

  • Approvals — Who can make changes and when. Is there an enforced workflow, or does it happen in Slack?
  • Policies — What is allowed or disallowed. Are standards documented but unenforced, or automatically applied?
  • Access Control — Who has permission to do what. Is access consistent across teams and environments?
  • Audit Trail — What happened, and can you prove it. How long does audit prep take today?
🔎 Vague Governance Requests
"We need better controls"
"Approvals aren't consistent"
"Security is getting more involved"

These are broad statements. Break them into specifics — which of the four areas is actually broken?

02How Governance Works Today — the Reality
▼

Most teams govern infrastructure the same way — manually and inconsistently. This works early. At scale, it fails because it relies on people instead of systems.

How approvals happen

Slack messages, Jira tickets, manual reviews. No enforced workflow. Depends entirely on people following the process.

How audit evidence is gathered

CI logs, ticket histories, screenshots. Manual, time-consuming, and incomplete.

Why This Fails at Scale

Manual governance is inconsistent across teams, difficult to audit and enforce, and breaks down as the number of engineers and environments grows. The intention exists — the enforcement doesn't.

03The Four Ways Governance Breaks
▼

1. Approvals Are Manual and Inconsistent

🔎 Signals
"Approvals are done in tickets"
"We review things manually before prod"

Impact: Risk of unreviewed changes reaching production, slower deployments. Ask: "How are production changes approved today? Is that consistent across teams?"

2. Policies Exist But Aren't Enforced

🔎 Signals
"We have policies, but they're not always followed"
"Standards vary by team"

Impact: Drift from standards, increased risk exposure. Ask: "Are policies enforced automatically or reviewed manually? What happens when a team violates a standard?"

3. Access Control Is Fragmented

🔎 Signals
"Access is inconsistent across teams"
"Permissions are hard to manage"

Impact: Security risk, potential over-permissioning. Ask: "How is access to infrastructure controlled today? Is it consistent across environments?"

4. Audit Trail Is Weak or Manual

🔎 Signals
"Audit prep is painful"
"We have to piece together logs and tickets"

Impact: Time-consuming audits, lower confidence in controls. This often drives budget and urgency. Ask: "How do you prove who changed what in infrastructure? How long does audit preparation take?"

04When Governance Becomes Urgent — and Gets Budget
▼

Governance problems often don't start the buying process. They become urgent — and funded — when triggered by a specific event.

  • Security audit with a deadline attached
  • Enterprise customer requiring compliance certification
  • Production incident traced back to an unreviewed change
  • Compliance deadline — SOC 2, ISO 27001, etc.
🔎 High-Urgency Signals
"We have an audit coming up"
"Security flagged this"
"We need to improve controls"
Why This Gets Budget

When security or compliance is involved, this isn't a backlog item. There's typically a deadline, a named stakeholder, and budget already allocated. These conversations accelerate.

05Pattern Recognition Framework
▼
  • Hear the signal — "Approvals are manual" / "Audit prep is painful" / "Security is getting more involved"
  • Interpret — Governance is inconsistent; controls are not enforced; risk exposure exists
  • Expand — "Where does the process break down? What is hardest to prove or enforce?"
  • Identify type — Control problem? Audit problem? Policy enforcement gap?
  • Light positioning (only after confirming pain) — "Most teams at that stage are trying to move from manual, process-driven governance to something enforced directly in the infrastructure workflow."
06What Good Discovery Sounds Like
▼

✗ Weak

"Do you need better governance tools?"

✓ Strong

"How are infrastructure changes approved today? How do you enforce policies across teams? How do you prove changes for audits?"

The Reframe

"It sounds like your governance model relies heavily on manual processes, which is becoming harder to manage as you scale."

▼

MODULE 05 / 06

Platform Teams vs Developer Velocity

This isn't just about tools — it's about how the company operates. The platform bottleneck is one of the highest-probability entry points for a Spacelift conversation.

⏱ ~8 min8 sectionsKnowledge check
01The Core Tension: Speed vs. Control
▼

As companies grow, infrastructure responsibility centralizes into a platform team. Their job: build and maintain infrastructure, enforce standards, keep production safe. At the same time, product teams want to move faster and more infrastructure requests keep coming.

What Developers Want

Speed and autonomy. Ship features without waiting. Provision what they need without filing a ticket.

What Platform Teams Need

Control and guardrails. Consistent standards. Production stays safe. Security and compliance requirements met.

Key Takeaway

Every infrastructure conversation at scale is shaped by this tension. When you hear friction between platform and development teams, you're hearing the same underlying problem.

🔎 Organizational Friction Signals
"Platform team is overloaded"
"Developers are blocked on infrastructure"
"Everything has to go through platform"
02What It Looks Like in the Real World
▼

The default model: developers need infrastructure, they submit a ticket, platform reviews and implements, changes go through approval. At small scale, this works. At scale, it breaks.

  • Request volume increases faster than platform team capacity
  • Platform team becomes a bottleneck for every team that needs infrastructure
  • Delivery slows down — developers waiting days for something that should take minutes
  • Frustration builds on both sides
🔎 Bottleneck Signals
"We have too many requests"
"Platform can't keep up"
"Everything is stuck in a queue"
03The Two Failed Extremes
▼

Organizations usually try one of two things when the bottleneck becomes painful. Neither fully solves the problem.

Extreme 1: Lock Everything Down

Platform owns everything. Strict approvals. Limited developer access.

  • Platform team overloaded
  • Developers blocked
  • Slower product delivery

Extreme 2: Open Everything Up

Developers provision freely. Minimal governance. Little consistency.

  • Increased risk
  • Security and compliance concerns
  • Incidents from unreviewed changes
🔎 Two Failure Mode Signals
"Everything goes through platform" → Over-centralization
"Teams just do their own thing" → Lack of control

Most companies are stuck somewhere between these two extremes. That's your opening.

04The Desired State: Self-Service With Guardrails
▼

What mature teams are actively working toward: developers can provision infrastructure themselves, platform defines what's allowed, policies enforce the boundaries automatically.

  • Developers provision infrastructure without filing tickets
  • Platform defines standards once — not executes every request
  • Policies enforce automatically, not manually
  • Workflows are standardized across teams
🔎 Future State Signals — High Quality Entry Points
"We want self-service with guardrails"
"We need to reduce tickets to platform"
"We want consistency without slowing teams down"
Why This Gets Budget

Self-service with guardrails is often a leadership mandate, not just an engineering goal. When a prospect mentions an "internal developer platform" or "developer experience initiative," there's usually a sponsor above the platform team driving it.

05Pattern Recognition Framework
▼
  • Hear the signal — "Platform team is slammed" / "Developers have to file tickets for everything"
  • Interpret — Bottleneck exists; workflow doesn't scale; organizational friction is high
  • Expand — "What still requires platform involvement? Where are developers blocked most often?"
  • Identify type — Bottleneck problem? Operating model issue? Control vs. speed imbalance?
  • Light positioning (only after confirming pain) — "Most teams at that stage are moving toward a self-service model where developers provision within defined guardrails, instead of routing everything through platform."
06What Good Discovery Sounds Like
▼

✗ Weak

"We help with self-service infrastructure."

✓ Strong

"What types of infrastructure requests still go through your platform team today? Where do developers get blocked most often? What needs to stay controlled vs. delegated?"

The Reframe

"It sounds like your platform team is stuck between enabling developers and maintaining control — and the current model isn't scaling cleanly."

MODULE 06 / 06

Where Spacelift Fits — and When It Doesn't

Everything comes together here. Your job is not to pitch Spacelift — it's to recognize when a real problem exists and guide the conversation toward the right next step.

⏱ ~8 min9 sectionsKnowledge check
01The Core Insight — Spacelift Is Not a Tool Swap
▼

Spacelift is NOT

  • A replacement for Terraform
  • A better CI/CD tool

Spacelift IS

  • A system for managing how infrastructure changes are executed, governed, and coordinated at scale
Key Takeaway

If the problem is small or local → Spacelift is likely not the right conversation. If the problem is cross-team, workflow-related, governance-related, or scaling-related → Spacelift becomes relevant.

🔎 Surface-Level Conversations to Go Deeper On
"We're evaluating Terraform Cloud alternatives"
"We already have pipelines"

These are often entry points, not the real story. Understand the underlying problem before positioning anything.

02The Five Problem Types Spacelift Solves
▼

You don't need to memorize features. You need to recognize which problem type exists — and whether it's significant enough to act on.

  • Platform Bottleneck (Speed) — Platform team overloaded, developers blocked, requests slowing delivery
  • Governance & Audit (Control) — Manual approvals, audit pain, security team involvement, compliance pressure
  • System Complexity (Coordination) — Multi-team coordination issues, dependencies across systems, fragmented workflows
  • Security & Risk (Exposure) — Credential concerns, drift or unknown changes, inconsistent access control
  • Tooling & Migration (Change) — Terraform Cloud limitations, cost concerns, multiple tools across teams
03The Conversation Framework
▼
  • Open and ground — "Can you walk me through how infrastructure changes get deployed today?"
  • Diagnose the problem — Use signals from Modules 2–5 to identify the pattern
  • Expand the problem — "Where does this break down? Who is impacted? How often?"
  • Identify the problem type — Bottleneck? Governance? Coordination? Security? Tooling?
  • Light positioning (only after confirming pain) — "Most teams at that stage start to look for a more structured way to manage how infrastructure changes are run across teams."
  • Qualify the next step — Is this worth a deeper technical conversation?
04What Good Positioning Sounds Like
▼

✗ Weak

  • "We are an IaC orchestration platform."
  • "We replace Terraform Cloud."

✓ Strong

  • "It sounds like the challenge isn't Terraform itself, but how the workflow is coordinated across teams."
  • "That's usually where a more structured orchestration and governance layer becomes useful."
05When Spacelift Is a Strong Fit
▼
  • Multiple teams using Terraform
  • CI/CD pipelines managing infrastructure workflows
  • Platform team bottleneck or manual approval processes
  • Audit or compliance pressure from security team
  • Drift, visibility gaps, or unknown infrastructure changes
  • Cross-team coordination challenges
🔎 High-Probability Signals
"Things are breaking across teams"
"We don't have consistency"
"We're struggling to manage this at scale"

These indicate systemic problems. The buyer will likely see value in a platform-level solution, not a tool-level fix.

06When Spacelift Is NOT a Fit — and Why This Matters
▼
  • Very small team (1–3 engineers)
  • No dedicated platform or DevOps function
  • Minimal infrastructure complexity
  • No governance or compliance needs
  • Terraform is working fine without friction
The Credibility Play

Being transparent about non-fit builds long-term credibility. "It sounds like your current setup is working well for your scale. This typically becomes more relevant as teams and workflows grow more complex." A rep who said that is the first call they make in 18 months.

07Rep Decision Framework
▼

After every call, ask yourself these questions:

  • Is there a real problem — not just a theoretical one?
  • Is the problem recurring and impactful enough to act on?
  • Does it involve multiple teams, governance, or coordination?
  • Does it reflect one of the five core problem types?

All YES → Move forward

Earn the next step — suggest a deeper technical conversation.

Any NO → Don't force it

Disqualify respectfully. Stay in touch. They may grow into it.

"Approvals are manual"
"We run Terraform through CI/CD"
"Audit prep is painful"

MODULE 07 · CALL SKILLS

How to Run a Great First Call

A great first call doesn't sound like a pitch. It sounds like a conversation that uncovers something real.

⏱ ~8 min8 sectionsKnowledge check
01The Goal of the First Call
▼

What you are NOT trying to do

  • Give a full product pitch
  • Explain every feature
  • Close a deal

What you ARE trying to do

  • Understand how infrastructure works today
  • Identify if a real problem exists
  • Determine if it's worth a deeper conversation
Definition of Success

Clear problem identified. Next step earned. That's it.

02The Call Structure
▼
  • Open (2–3 min) — Set context, build rapport, frame the conversation. "Would it be helpful to start with how you're currently managing infrastructure changes today, and then we can see where it might make sense to go deeper?"
  • Understand the system (5–7 min) — Map Code → Git → Pipeline → Cloud. "How do infrastructure changes get deployed today? What tools are involved?"
  • Find the friction (8–12 min) — Identify where things break. Look for problem patterns from Modules 2–5. "Where does that process start to break down?"
  • Expand the problem (5–8 min) — Make it real and impactful. "How often does that happen? Who does it affect? What happens when it goes wrong?"
  • Light positioning (2–3 min) — Only after confirming pain. "Most teams at that stage start to look for a more structured way to manage how infrastructure workflows are coordinated."
  • Close for next step (2–3 min) — "If it makes sense, we can bring in one of our solutions engineers to walk through how teams typically solve this."
03Strong Call vs. Weak Call
▼

✗ Weak Call

  • Jumps into product too early
  • Asks shallow tool-level questions
  • Accepts "it's messy" without digging
  • Doesn't identify real pain
  • Pushes for demo without confirmed problem

✓ Strong Call

  • Starts with system understanding
  • Finds real friction and expands it
  • Connects pain to business impact
  • Positions lightly and specifically
  • Earns a logical next step
04The Four Common Mistakes
▼

1. Pitching Too Early

"We're an IaC orchestration platform…" — Fix: ask 2–3 more questions before positioning anything.

2. Staying at Tool Level

"What tools are you using?" — Fix: move to workflow and coordination questions immediately.

3. Not Expanding the Problem

Accepting "it's messy" without digging deeper. Fix: always ask impact questions — how often, who, what happens when it goes wrong.

4. Forcing a Fit

Trying to push a demo with no clear problem. Fix: be willing to disqualify. A respectful "not yet" builds more pipeline than a forced demo.

05Real Call Example (Condensed)
▼

▼
  • Is there a real problem?
  • Is it recurring and impactful?
  • Does it involve multiple teams or workflows?
